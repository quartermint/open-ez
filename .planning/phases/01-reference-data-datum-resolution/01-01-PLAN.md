1. **Add field** `datum_offset_in` to the `GeometricParams` dataclass, in the fuselage stations section (after `fs_tail`):
   ```python
   # === DATUM OFFSET (internal -> published coordinate translation) ===
   datum_offset_in: float = 51.0  # Internal FS - published FS (see to_published_datum())
   ```

   The exact value: The code computes NP at FS ~153.5 (internal). Published Long-EZ NP is ~FS 108 (RAF CP-29). The offset is `153.5 - 108.0 = 45.5`. However, the CONTEXT.md and known issue say "~51 inches." The exact offset should be derived carefully:
   - Internal NP: 153.5 (from `calculate_neutral_point()`)
   - Published NP: 108.0 (from RAF CP-29, now in reference_data.json)
   - Offset = 153.5 - 108.0 = 45.5

   Use 45.5 as the initial value. Add a comment explaining:
   ```python
   datum_offset_in: float = 45.5  # Internal FS 153.5 maps to published FS ~108 (RAF CP-29)
   # Note: Previously estimated at ~51". Actual offset derived from NP comparison.
   # internal_fs - datum_offset_in = published_fs
   ```

2. **Add method** `to_published_datum()` on `GeometricParams`:
   ```python
   def to_published_datum(self, internal_fs: float) -> float:
       """Convert internal fuselage station to published Long-EZ datum.

       The internal coordinate system uses a different zero reference than
       the published Rutan plans. This method translates internal FS values
       to the published coordinate system for human-readable output.

       Args:
           internal_fs: Fuselage station in internal coordinates (inches)

       Returns:
           Fuselage station in published Long-EZ datum (inches)
       """
       return internal_fs - self.datum_offset_in
   ```

3. **Verify no regressions** — the field is additive only. No existing code references `datum_offset_in` yet. The `validate()` method on `AircraftConfig` should not need changes since this is a non-safety field.
