# Phase 03 Plan 06: Fix VSPAERO VLM Solver Pipeline

## Objective

Fix `_run_native_sweep()` so VSPAERO VLM actually solves and produces real aerodynamic data. Currently VSPAERO exits immediately with 0 thin surfaces and 0 Kutta nodes — the geometry export pipeline doesn't correctly classify lifting surfaces as thin VLM panels.

## Gap Source

- **03-VERIFICATION.md**: Truths #2 and #3 are HUMAN NEEDED — real polars never produced
- **.continue-here.md**: VSPAERO reports `NumberOfThinSurfaces_: 0`, `NumKuttaNodes: 0`, Z-coords ~1e15
- **Root cause**: The `ExportFile()` thin surface classification and/or VSPAEROSweep analysis configuration is incorrect for OpenVSP 3.48.2

## Known Issues (from debugging session)

1. `ExportFile(vspgeom_path, vsp.SET_ALL, vsp.EXPORT_VSPGEOM, False, THIN_SET)` — thick_set=SET_ALL includes wings as thick bodies, potentially conflicting with thin_set
2. Geometry Z-coordinates in .vspgeom are ~1e15 (should be small values in inches)
3. VSPAEROSweep analysis may need explicit `GeomSet` and/or geometry file path inputs
4. `vsp.Update()` is called before `SetSetFlag()` — may need re-update after set classification
5. Control surface parm errors suggest geometry setup issues

## Tasks

<task id="1" title="Create standalone VSPAERO diagnostic script">
**Goal:** Build a minimal script that isolates the VSPAERO pipeline from the full Long-EZ model. One simple rectangular wing + VSPAERO at a single alpha point. This reveals whether the issue is in geometry complexity or fundamental API usage.

**Steps:**

1. Create `scripts/vspaero_diagnostic.py`:
   - Import openvsp, clear model
   - Add a single WING with simple rectangular geometry (span=100, chord=20, no sweep/dihedral)
   - Call `vsp.Update()`
   - Create user set for thin surfaces, add wing to it
   - Call `vsp.Update()` again after set classification
   - Export VSPGEOM with:
     - Option A: `ExportFile(path, SET_NONE, EXPORT_VSPGEOM, False, thin_set)` — only thin surfaces, no thick
     - Option B: `ExportFile(path, SET_ALL, EXPORT_VSPGEOM, False, thin_set)` — current approach
   - Print file size and check for reasonable coordinate values
   - Run VSPAEROSweep at alpha=5 degrees
   - Print result arrays (CL, CD)
   - Include diagnostic prints at every step: number of geoms, set membership, file existence, VSPAERO stdout

2. The script should be self-contained and runnable with `python3.13 scripts/vspaero_diagnostic.py`

3. Print all available VSPAEROSweep input names using `vsp.GetAnalysisInputNames("VSPAEROSweep")` to discover what parameters the solver accepts in OpenVSP 3.48.2

**Verification:** Script runs without crash and prints diagnostic info showing whether VSPAERO iterates or exits immediately. Output reveals which API approach produces thin surfaces.

**CHECKPOINT:** After running the diagnostic, report findings before proceeding to Task 2. The fix approach depends entirely on what the diagnostic reveals.
</task>

<task id="2" title="Fix _run_native_sweep() based on diagnostic findings">
**Goal:** Apply the working API pattern from Task 1's diagnostic to `core/vsp_integration.py`'s `_run_native_sweep()` method.

**Likely fixes (based on known issues — actual fix depends on Task 1 findings):**

1. **Thin/thick set separation**: Instead of `thick_set=SET_ALL`, use a separate thick set containing only the fuselage. Wings in both SET_ALL (thick) and THIN_SET may cause VSPAERO to treat them as thick displacement bodies.

2. **Second `vsp.Update()` after set flags**: Call `vsp.Update()` after `SetSetFlag()` calls at lines 196-199, before `ExportFile()` at line 205.

3. **Explicit VSPAEROSweep geometry inputs**: Set `GeomSet` and/or VSPGEOM file path in the analysis inputs if discovered by diagnostic.

4. **Geometry coordinate validation**: If Z-scaling issue is from OpenVSP internals, may need to adjust geometry build order or parameter values.

**Steps:**
1. Read Task 1 diagnostic output to identify the working pattern
2. Apply fix to `_run_native_sweep()` in `core/vsp_integration.py`
3. Run `python3 -m pytest tests/ -x -q` to confirm no regressions (196+ tests pass)
4. Run `python3.13 main.py --analysis` to confirm native VSPAERO path triggers and solver iterates

**Verification:**
- `python3.13 main.py --analysis` prints "Using native VSPAERO" and VSPAERO iterates (not immediate exit)
- CL values are non-zero and physically plausible (CL ~0.0 at alpha=0, increasing with alpha)
- All existing tests still pass
</task>
